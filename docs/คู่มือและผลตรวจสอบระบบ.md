# ระบบจัดการเอกสาร UAT / UAI — “Project Document” (Document ES)

เอกสารฉบับนี้ประกอบด้วย 3 ส่วน
1. **ผลการตรวจสอบระบบ & จุดบกพร่อง** (สิ่งที่แก้ไปแล้ว + สิ่งที่แนะนำให้ทำต่อ)
2. **คู่มือการใช้งาน**
3. **ข้อดี – ข้อเสีย / ข้อจำกัด**

> ตรวจสอบโดยรีวิวโค้ดทั้ง Backend (Go/Gin/pgx/PostgreSQL) และ Frontend (Next.js/TypeScript/Tailwind/React Query) แบบละเอียด

## ส่วนที่ 1 — ผลการตรวจสอบระบบ

### 1.1 ภาพรวม
โครงสร้างระบบ **แข็งแรงและเขียนอย่างระมัดระวัง**: ใช้ transaction + `defer rollback`, ลบไฟล์คืนเมื่อ tx ล้มเหลว, query แบบ parameterized ทั้งหมด (กัน SQL injection), มี `isAdmin()` ป้องกันทุก route ที่แก้ไขข้อมูล, refresh token เก็บแบบ hash + เช็คเพิกถอน, และ Acknowledge ใช้ DB lock (`ON CONFLICT`) กัน race — ตรวจแล้ว**ไม่พบช่องโหว่ RBAC หลุด**

> ⚠️ **ข้อมูลเดิมที่ได้รับผลกระทบจากข้อ 1:** เอกสารที่เคยกรอกผ่านฟอร์มช่วงที่บั๊กอยู่ อาจมีค่า UAT/UAI สลับกันในฐานข้อมูล — ระบบ**แก้ย้อนหลังอัตโนมัติไม่ได้** (ไม่รู้เจตนาเดิม) แนะนำให้ admin เข้าไปตรวจ/แก้เอกสารที่ค่า UAT ≠ UAI ด้วยตนเอง (เอกสารที่ทั้งสองสถานะเหมือนกันไม่ได้รับผลกระทบ)

### 1.3 จุดที่ “แนะนำให้ปรับต่อ” (ยังไม่แก้ในรอบนี้)

| ระดับ | เรื่อง | คำอธิบาย / คำแนะนำ |
|---|---|---|
| ✅ **แก้แล้ว** (2026-07-07) | ~~**SSE ใช้ token ค้าง** (`NotificationBell`)~~ | เดิม: stream เปิดด้วย access token ตอน mount; หลัง refresh สายเดิมยังใช้ token เก่า → realtime หยุดจนกว่าจะ reload — **แก้แล้ว:** เพิ่ม pub/sub ใน `token.ts` (`subscribe`/`notify` ใน set/setAccess/clear) + `NotificationBell` อ่าน token สดทุกครั้งที่ connect และ reconnect เมื่อ token เปลี่ยน (loop: poll 60 วิ → 401 → refresh → notify → reconnect) |
| ✅ **แก้แล้ว** (2026-07-07) | ~~**ดาวน์โหลดไฟล์ข้าม interceptor**~~ | เดิม: ปุ่มดาวน์โหลดใช้ `fetch` แนบ token ตรงๆ ไม่ผ่าน axios → token หมดพอดีดาวน์โหลดไม่ได้ ไม่ retry — **แก้แล้ว:** เพิ่ม `documentApi.download()` (axios `responseType:"blob"`) + helper `saveBlob()` ใน `utils.ts`, เปลี่ยนทั้ง 3 จุด (list/detail/edit) ให้ผ่าน interceptor (401→refresh→retry) |
| ✅ **แก้แล้ว** (2026-07-07) | ~~**`SearchableSelect` reset query**~~ | เดิม: effect เปิด dropdown อ้างอิง `options` (array ใหม่ทุก render) → parent re-render ล้างคำค้นหา — **แก้แล้ว:** gate ด้วย `wasOpen` ref ให้ reset เฉพาะตอน false→true (เพิ่งเปิด) เท่านั้น |
| 🔵 ข้อควรพิจารณา (security) | **เก็บ JWT ใน `localStorage`** | อ่านได้ด้วย XSS และอยู่ข้ามแท็บ/รีสตาร์ท — ทางเลือกปลอดภัยกว่าคือเก็บ refresh token เป็น httpOnly cookie |
| 🔵 ข้อควรพิจารณา | **ดาวน์โหลดไม่มีสิทธิ์ระดับเอกสาร** | ผู้ล็อกอินคนใดก็ดาวน์โหลดไฟล์เอกสารใดก็ได้ (เป็น design “Global — เห็นทุกเอกสาร”) — ยืนยันว่าตั้งใจ |
| 🔵 ข้อควรพิจารณา | **CORS + AllowCredentials** | อนุญาต private-LAN แบบกว้าง — ตอนขึ้น production ควรตั้ง `CORS_ALLOWED_ORIGINS` ให้ชัดเจน |
| 🔵 ข้อจำกัดเชิงออกแบบ | **Acknowledge เป็น single-lock** | 1 เอกสาร มีผู้กด “รับทราบ” ได้เพียง **คนเดียว** (first-come) แม้ป้ายจะเขียน “Multi-user Acknowledge” — ถ้าต้องการให้หลายคนรับทราบต่อเอกสาร ต้องปรับ schema (ลบ unique constraint `uq_acknowledgements_document`) |
| 🔵 พฤติกรรม | **เอกสารไม่มีไฟล์ กดรับทราบไม่ได้** | เพราะ ack อ้างอิง version ล่าสุด — ต้องมีไฟล์อย่างน้อย 1 ก่อนถึงจะรับทราบได้ |
| 🔵 Ops | **รันด้วย `go run` (dev)** | โหมด dev ไม่ hot-reload และต้อง restart เอง — ตอน production ควร build เป็น binary + ใช้ process manager (systemd / NSSM) และตั้ง env `APP_ENV=production` |

### 1.4 จุดที่ตรวจแล้ว “ทำได้ดี” 👍
- `isAdmin()` ครอบทุก route ที่แก้ไขข้อมูล — ไม่มีช่องหลุด
- SQL parameterized ทั้งหมด; query dynamic เดียว (Update เอกสาร) สร้างชื่อคอลัมน์จาก whitelist ไม่ใช่ input ผู้ใช้
- Transaction + cleanup ไฟล์เมื่อ fail ครบทุกจุด (create/add/delete)
- Refresh token: hash + expiry + revocation + เพิกถอน session ตอนปิดบัญชี
- ป้องกัน admin ล็อกเอาต์ตัวเอง (ปิด login ทุกวิธีไม่ได้)
- Loading/Empty state ครบทุกตาราง; error extraction pattern สม่ำเสมอ
- Accessibility พื้นฐาน: `role="dialog"`, `aria-label`, จัดการ Esc/Enter
- axios interceptor ทำ single-flight refresh (กัน refresh ซ้อน)

## ส่วนที่ 2 — คู่มือการใช้งาน

### 2.1 บทบาทผู้ใช้ (Roles)
- **Admin** (`SystemAdmin`/`admin`): เห็นทุกเมนู, เข้า **Settings**, แก้ไข/ลบเอกสาร, จัดการ Master & ผู้ใช้
- **User** ทั่วไป: เห็นเมนูตามที่ admin เปิด, อัปโหลด/ดู/ดาวน์โหลด/กดรับทราบเอกสารได้ แต่**แก้ไข/ลบไม่ได้**

### 2.2 การเข้าสู่ระบบ (`/login`)
- **Local Login**: username + password ที่ admin สร้าง
- **Microsoft (Azure AD)**: ล็อกอินด้วยบัญชี Microsoft (ถ้า admin เปิด) — user ใหม่ได้ Role: User อัตโนมัติ
- admin เปิด/ปิดวิธี login ได้ที่ Settings → การเข้าสู่ระบบ (ปิดทุกวิธีไม่ได้)

### 2.3 Dashboard (`/dashboard`)
- **การ์ด KPI**: เอกสารทั้งหมด / ของฉัน / รอรับทราบ / รับทราบวันนี้
- **กราฟ “เอกสารต่อวัน”**: เลือกช่วง 7/14/30 วัน หรือกำหนดเอง (ปฏิทิน from–to), hover ดูตัวเลขรายวัน
- **โดนัท “สถานะ UAT/UAI”**: จำนวนเอกสารแยกตามสถานะ Pending/Passed/Failed — ชี้ที่วง/แถว legend เพื่อเน้นสถานะ
- **กิจกรรมล่าสุด**: รวม 3 เหตุการณ์ — 🔵 อัปโหลดใหม่ / 🟢 กดรับทราบ / 🟡 แก้ไขเอกสาร (แสดง 5 รายการ เกินนั้นเลื่อน scroll ในกล่อง)

### 2.4 Documents (`/documents`)
**ตารางรายการ** — คอลัมน์: บริษัท / ประเภท / วันที่ติดตั้ง / WorkOrder / เจ้าหน้าที่ / สถานะ+วันที่ UAI / สถานะ+วันที่ UAT / ไฟล์ / รับทราบ / จัดการ(admin)
- **ค้นหา** (บริษัท/WO/ผู้รับผิดชอบ) ที่ช่องบนขวา
- **กรอง/เรียงต่อคอลัมน์**: คลิกหัวคอลัมน์ → เลือกค่า (checkbox หลายค่า) + เรียงน้อย→มาก/มาก→น้อย (รวมหลายคอลัมน์แบบ AND) → footer มีปุ่มล้างตัวกรอง
- **ไฟล์**: แสดงไอคอนตามชนิด (PDF = แดง, ไฟล์อื่น = ฟ้า) — คลิกไอคอนเพื่อดาวน์โหลด, hover เห็นชื่อไฟล์
- **รับทราบ**: ปุ่มไอคอน — กดแล้ว**ล็อกถาวร** (1 เอกสาร/1 คน)
- **จัดการ (admin)**: ✏️ แก้ไข / 🗑️ ลบ (ลบถาวร + ลบไฟล์จริง มี confirm)

**เพิ่มเอกสาร (`/documents/upload`)** — 2 คอลัมน์: ซ้าย = ข้อมูล Project + สถานะ UAT/UAI, ขวา = แนบไฟล์
- **ประเภท**: dropdown (Standard/Modify/Add-on)
- **บริษัท**: พิมพ์ค้นหา/เลือกจาก Master หรือ**พิมพ์ชื่อใหม่**ได้ (มีแถว “＋ ใช้ …”)
- **ผู้รับผิดชอบ**: dropdown ค้นหาได้ (จาก Master พนักงาน)
- **สถานะ UAT/UAI**: ถ้าไม่เลือก = ไม่กำหนด (แสดง “—”, ไม่บังคับเป็น Pending)
- **ไฟล์**: PDF/JPG/PNG ≤ 10MB ต่อไฟล์ (ไม่บังคับแนบ) — ปุ่ม บันทึก(ซ้าย)/ยกเลิก(ขวา) มุมขวาบน
- **แจ้งเตือน**: เมื่อบันทึกโดย UAT หรือ UAI = **Passed** ระบบจะแจ้งเตือนทุกคน

**แก้ไข (admin, `/documents/[id]/edit`)**: ฟอร์มเดียวกับหน้าเพิ่ม + จัดการไฟล์ (เพิ่ม/ลบไฟล์ได้ทันที) — บันทึกแล้วมี Toast; แจ้งเตือนเมื่อสถานะ**เพิ่งเปลี่ยนเป็น Passed**

**รายละเอียด (`/documents/[id]`)**: ข้อมูล project, สถานะ, รายการไฟล์ (ดาวน์โหลด), รายชื่อผู้รับทราบ

### 2.5 Settings (admin เท่านั้น, `/settings`)
- **สิทธิ์เมนู**: เปิด/ปิดเมนูที่ Role User เห็น
- **การเข้าสู่ระบบ**: เปิด/ปิด Local / Azure AD + ตั้ง Tenant/Client ID
- **ผู้ใช้งาน**: แก้ไขชื่อ/รหัสพนักงาน/ระดับ/แผนก/Role/สถานะ, ปิดบัญชี (soft-delete)
- **Master**: จัดการ **แผนก / ระดับ / รายชื่อพนักงาน / บริษัท** (ใช้เป็น dropdown ในหน้า Upload) — ทุกการบันทึกมี Toast แจ้งผล

### 2.6 การแจ้งเตือน (🔔)
- realtime ผ่าน SSE + poll สำรองทุก 60 วิ
- ชนิด: **สถานะ Passed** (แจ้งทุกคน) / **มีผู้รับทราบ** (แจ้งผู้อัปโหลด) — คลิกเพื่อไปยังเอกสาร, กด “อ่านทั้งหมด” ได้

## ส่วนที่ 3 — ข้อดี / ข้อเสีย

### ✅ ข้อดี
- **ความปลอดภัยแน่นหนา**: RBAC 2 ชั้น (frontend ซ่อน + backend บังคับ), parameterized SQL, JWT + refresh rotation
- **ครบวงจร UAT/UAI**: อัปโหลด → ติดตามสถานะ → รับทราบ → ประวัติ ในที่เดียว
- **UX ดี**: Toast แจ้งผล, dropdown ค้นหาได้, filter/sort ต่อคอลัมน์, กราฟ interactive, sidebar ย่อ/ขยาย + responsive มือถือ
- **ข้อมูลกลาง (Master) จัดการเองได้** โดย admin — ยืดหยุ่น
- **Realtime notification** + fallback ที่ทนต่อการหลุดของ stream
- **โครงสร้างโค้ดสะอาด** แยก layer ชัด, migration เป็นระบบ (auto-run + tracked)

### ⚠️ ข้อเสีย / ข้อจำกัด
- **Acknowledge ได้คนเดียวต่อเอกสาร** (single-lock) — ไม่ตรงกับคำว่า “Multi-user” ถ้าต้องการหลายคนต้องแก้ schema
- **โหลดรายการเอกสารทีละ 200 แถว** (frontend ไม่มี pagination จริง) — ถ้าเอกสารเยอะมากจะช้า/กินหน่วยความจำ ควรทำ paging/infinite scroll
- **JWT อยู่ใน localStorage** — เสี่ยง XSS ขโมย token (ทางเลือก: httpOnly cookie)
- **Dashboard ยิงหลาย query ต่อครั้ง** (~13) — โอเคที่ปริมาณน้อย แต่ควรรวม query ถ้าโตขึ้น
- **รันแบบ dev (`go run`)** — ต้อง restart เองเมื่อแก้โค้ด; ยังไม่มี process manager สำหรับ production
- **Redirect URI / origin เป็นค่า localhost/LAN** — ต้องปรับตอน deploy จริง
- **แก้ไขซ้ำหลายครั้ง** จะขึ้นในกิจกรรมล่าสุดหลายบรรทัด (ยังไม่ dedupe)
